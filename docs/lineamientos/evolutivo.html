¿Qué es un Sprint?​

Un Sprint es como una carrera corta que dura una semana. Durante este tiempo, un equipo trabaja para completar tareas importantes y mejorar algo que están construyendo (como un juego o una app). Antes de empezar, se hace un plan para saber qué cosas se tienen que hacer y cómo lograrlas.

¿Cómo se planifica un Sprint?​

  • Hablar sobre las tareas: El equipo se reúne con el Project Manager (PM). Juntos deciden qué cosas son más importantes para trabajar.
  • Estas tareas deben decidirse antes del día 10 de cada mes para aprovechar bien el tiempo.
  • Si alguien pide algo después del día 20, el equipo tiene que ver si es posible incluirlo o si debe esperar al próximo mes.
  • Tiempo limitado: El tiempo que tiene el cliente (la persona o empresa que pide el trabajo) no se guarda para el mes siguiente, así que todas las tareas deben hacerse en el mismo mes.
  • Cómo hablar con el equipo:
    • Las consultas rápidas se hacen por WhatsApp.
    • Las decisiones importantes y aprobaciones se hacen por correo electrónico, para que quede todo claro y registrado en Wrike. Todo aquello que no se encuentre registrado en Wrike será ignorado.

¿Qué es el Product Backlog?​

El Product Backlog es como una lista de deseos. Aquí se escriben todas las ideas y tareas que el cliente necesita. Es el trabajo del PM asegurarse de que esta lista esté actualizada y organizada.

Pasos para organizar el Product Backlog:​

  1. Hacer una lista de tareas:
    El PM se reúne con el cliente para entender qué cosas son importantes para trabajar. Esto se hace al inicio del mes.
  2. Definir prioridades:
    Cada tarea se anota en una tabla especial (un tablero Scrum) con información como:
    • Nombre y descripción de la tarea.
    • Qué tan importante es.
    • Cuánto tiempo llevará hacerla.
  3. Calcular el esfuerzo:
    El PM y un experto técnico calculan cuántas horas llevará hacer cada tarea. Si algo no está claro, vuelven a hablar con el cliente para ajustar los detalles.
  4. Revisar todo con el cliente:
    Cuando todas las tareas están listas y organizadas, el PM se reúne otra vez con el cliente. Juntos deciden cuáles tareas se trabajarán en el próximo Sprint.

¿Cómo se manejan las tareas?​

Las tareas se escriben en un sistema llamado Wrike. Cada tarea tiene información como:

  • Qué se necesita hacer.
  • Requisitos (lo que debe estar listo antes de empezar).
  • Qué significa que la tarea esté bien hecha (criterios de aceptación).

El equipo también se asegura de tener toda la información técnica necesaria, como accesos a sistemas, para que puedan trabajar sin interrupciones.

Crear una Carpeta Cada Mes​

Cada mes, se crea una nueva carpeta y se llama con el nombre del mes, como .......................................(Cosentino - Abono 2025.01). En Wrike, los Project Managers (PM) deben crear una carpeta nueva al principio de cada mes. Esa carpeta servirá para anotar todo lo que pase ese mes con el cliente.

Tarea: "Gestión del Abono"​

Dentro de esa carpeta, hay que hacer una tarea especial que se llamará “Gestión del Abono”. Esta tarea es como una lista donde se escriben todas las horas que el equipo (los PM y los desarrolladores) dedicaron a organizarse y trabajar con el cliente.

¿Qué se Anota en la Tarea?​

En esa lista se escriben actividades como:

  • Reuniones: Cuando el equipo se junta para planear el trabajo o hablar con el cliente.
  • Notas de las Reuniones: Escribir lo más importante de lo que se habló.
  • Organizar las Tareas del Mes: Decidir qué cosas son más importantes y planear cómo se harán.
  • Resolver Preguntas: Tiempo que el equipo usa para responder dudas o hacer cambios que pide el cliente.
  • Otras Cosas de Organización: Cualquier cosa extra que ayude a que todo salga bien.

Mantener Todo Ordenado​

El PM tiene que asegurarse de que esta lista esté siempre actualizada. Así, todos saben cuánto tiempo usaron en organizar y trabajar con el cliente.

Estados de las Tareas​

  • Nueva: Alguien escribe una nueva tarea. El PM revisa los detalles y la envía al equipo técnico.
  • En análisis: El equipo técnico decide cuánto tiempo tomará hacer esa tarea. Debe tener en cuenta tiempos de QA e implementación en ambientes dev y producción, no solo horas de desarrollo.
  • Feedback del cliente: El PM habla con el cliente para que apruebe la tarea y decida si es importante o no.
  • En Backlog: Las tareas aprobadas se organizan para empezar a trabajarlas en el siguiente sprint.
  • En curso: Un desarrollador trabaja en la tarea. Si necesita más información, el equipo debe procurar conseguir esta información con el cliente (PM, desarrollador, QA, etc.).
  • Testing: El PM y el desarrollador revisan con el cliente si la tarea quedó bien hecha.
  • Lista para producción: La tarea se prepara para ser implementada en el ambiente productivo una vez esté el OK del cliente.
  • Completada: El trabajo está terminado y listo para que todos lo usen en ambiente productivo y con el OK del usuario.

Revisiones y Garantías​

Después de que algo nuevo se lanza, el cliente tiene dos semanas para probarlo. Si algo no funciona como se esperaba, deben:

  1. Avisar al PM y mostrar pruebas del problema.
  2. El equipo revisará qué pasó y arreglará los problemas en el próximo Sprint.

Tickets de Soporte​

Todos deben trabajar juntos para asegurarse de que cada problema ("ticket de soporte") sea resuelto de la mejor manera posible.

¿Qué pasa con los Tickets de Soporte?​

  1. Nuevo Pedido (Requerimiento): Crear una nueva tarea en el Product Backlog y cerrar el ticket de soporte.
  2. Problema Técnico (Bug): Priorizar con el cliente y corregir el problema.
  3. Fuera de Alcance: Asignar el ticket al equipo correcto y monitorear su resolución.
  4. Mal Asignado: Ajustar el sistema para evitar futuros errores.

En Resumen​

Cada mes, el equipo planifica tareas en una lista llamada Product Backlog. Estas tareas se trabajan en periodos cortos llamados Sprints. El PM organiza todo para que el cliente, los desarrolladores y el equipo técnico trabajen bien juntos. Si algo no sale perfecto, se ajusta y se mejora en el siguiente Sprint.
